# 02 Kolmogorov 复杂度：什么是最短描述？

问下大家，如果要你用一段程序生成字符串 `"abababababababababab"`，你会怎么写？

最笨的方法可能是：
```python
print("abababababababababab")
```

但聪明的你可能会写：
```python
print("ab" * 10)
```

第二种写法更短，对吧？这就是 **Kolmogorov 复杂度** 的核心思想：**用最短的程序来描述一个对象**！

## 什么是 Kolmogorov 复杂度？

**定义**：字符串 $x$ 的 Kolmogorov 复杂度 $K(x)$，是输出 $x$ 的最短程序的长度。

形式上：
$$K(x) = \min_{p} \{|p| : U(p) = x\}$$

其中：
- $p$ 是一个程序
- $|p|$ 是程序的长度（比特数）
- $U$ 是一个通用图灵机（Universal Turing Machine）
- $U(p) = x$ 表示程序 $p$ 在 $U$ 上运行输出 $x$

### 举个栗子 🌰

**例 1**：字符串 `"000...0"`（10000个0）
```python
# 程序 A（笨方法）
print("0" * 10000)  # 需要 10000 个字符

# 程序 B（聪明方法）
print("0" * 10000)  # 只需要 log₂(10000) ≈ 14 比特来编码数字10000
```

$K(\mathtt{0}^{10000}) \approx \log_2(10000) + O(1) \approx 14$ 比特

**例 2**：字符串 `"abababab..."`（10000个"ab"重复）
```python
print("ab" * 5000)
```

$K(\mathtt{ab}^{5000}) \approx \log_2(5000) + O(1) \approx 13$ 比特

**例 3**：随机字符串（无规律）
```python
print("xJ#9mK$pL2vN...")  # 只能硬编码
```

$K(\text{随机字符串}) \approx n$（字符串长度）

## 不可计算性：一个令人沮丧的事实

等等，既然 Kolmogorov 复杂度这么有用，我们是不是可以写一个程序来自动计算它？

**坏消息：Kolmogorov 复杂度是不可计算的！**

### 证明（反证法）

假设存在一个程序 $P$，可以计算任意字符串的 Kolmogorov 复杂度：

```
P("000...0") = 14
P("ababab...") = 13
P("随机串") = 10000
```

现在考虑下面这个程序 $Q$：

```
程序 Q(n):
    对于所有长度为 n 的字符串 s:
        如果 P(s) > n/2:
            返回 s
```

这个程序做了什么？它找到了一个 Kolmogorov 复杂度大于 $n/2$ 的字符串。

但是等等！程序 $Q$ 本身的长度大约是 $|P| + \log_2(n) + O(1)$，远小于 $n/2$（当 $n$ 很大时）。

这就产生了矛盾：我们找到了一个程序，它的长度小于 $n/2$，但却输出了一个 Kolmogorov 复杂度大于 $n/2$ 的字符串！

**结论**：假设错误，$P$ 不存在。Kolmogorov 复杂度是不可计算的。

### 这意味着什么？

1. **没有通用压缩算法**：不存在一个程序能找到所有字符串的最短描述
2. **但我们可以近似**：gzip、LZMA 等压缩算法提供了实用的近似
3. **理论上完美，实践中有挑战**

## 算法随机性：什么是真正的随机？

Kolmogorov 复杂度给了我们一个定义"随机性"的新视角：

**定义**：一个长度为 $n$ 的字符串 $x$ 是**算法随机的**（algorithmically random），如果：
$$K(x) \geq n - c$$

其中 $c$ 是一个常数。

换句话说：**真正的随机字符串，就是那些不能被显著压缩的字符串！**

### 有趣的事实

1. **大多数字符串是随机的**：长度为 $n$ 的字符串中，至少 $1 - 2^{-k}$ 的比例满足 $K(x) \geq n - k$
2. **但我们无法证明某个特定字符串是随机的**（因为 $K$ 不可计算）
3. **伪随机数生成器**：计算机生成的"随机"数其实都有很短的描述（种子+算法）

## 与香农熵的关系

Kolmogorov 复杂度和香农熵有什么关系？

| 特性 | 香农熵 | Kolmogorov 复杂度 |
|------|--------|------------------|
| 定义基础 | 概率分布 | 算法描述 |
| 计算难度 | 可计算（知道分布即可） | 不可计算 |
| 适用范围 | 随机变量 | 单个对象 |
| 典型值 | $H(X)$ | $\approx n$（对随机串） |

**联系**：对于从分布 $D$ 中随机抽取的字符串 $x$，有：
$$K(x) \approx -\log_2 D(x) \approx n \cdot H(D)$$

## 实际应用：用压缩近似 KC

虽然 Kolmogorov 复杂度不可计算，但我们可以用压缩算法来近似：

```python
import zlib

def approximate_kc(s):
    """用 gzip 压缩长度近似 Kolmogorov 复杂度"""
    compressed = zlib.compress(s.encode())
    return len(compressed)

# 测试
s1 = "0" * 10000      # 简单重复
s2 = "ab" * 5000      # 简单模式
s3 = "随机字符串..."   # 真随机（假设）

print(f"s1 的近似 KC: {approximate_kc(s1)}")
print(f"s2 的近似 KC: {approximate_kc(sc(s2)}")
print(f"s3 的近似 KC: {approximate_kc(s3)}")
```

**注意**：压缩算法只能识别特定类型的模式（如重复、频率统计），不能识别所有可压缩性。

## 小结

问下大家，现在理解 Kolmogorov 复杂度了吗？

**核心要点**：
1. $K(x)$ = 输出 $x$ 的最短程序长度
2. **不可计算性**：不存在通用算法能计算任意字符串的 KC
3. **算法随机性**：$K(x) \approx n$ 的字符串是真正随机的
4. **实用近似**：用 gzip 等压缩算法可以近似 KC

**为什么这很重要？**

Kolmogorov 复杂度为我们提供了一个**客观的复杂度度量标准**，不依赖于特定的表示方法或先验知识。它是信息论、机器学习、统计推断的基石。

但是，对于确定性系统，KC 有一个问题：如果系统演化是确定性的，那么 $K(\text{状态}_t) \leq K(\text{初始状态}) + \log_2(t) + O(1)$，只能对数增长，而不是我们直觉中的线性或多项式增长！

这就引出了下一篇要讲的：**资源受限 Kolmogorov 复杂度**。
